## Measured Prompt-Set Result

TensorRT-LLM prompt-set generation completed for all 50 fixed prompts.

Important limitation: this run used the TensorRT-LLM `examples/run.py` fallback once per prompt. That is valid for producing TensorRT outputs for quality comparison, but it includes process and engine startup overhead and must not be used as an optimized runtime speedup claim.

Key metrics from `trtllm_prompt_summary.json`:

- Backend used: `examples`
- Total requests: `50`
- Average latency: `21982.832 ms`
- P50 latency: `21854.962 ms`
- P95 latency: `22686.289 ms`
- Generation throughput: `3.777 tokens/sec`
- Max observed GPU memory: `40.51 GB`

The TensorRT-LLM LLM API path did not load the existing engine cleanly:

```text
Unable to initialize TensorRT-LLM LLM API:
llmapi_engine_backend:
llmapi_engine: Executor worker returned error
llmapi_model_with_engine: LLM got invalid argument: engine_dir
```

Observed root issue in logs:

```text
Unrecognized model in artifacts/trtllm_engine_bf16_or_fp16. Should have a `model_type` key in its config.json.
```

Conclusion: for this TensorRT-LLM container and engine layout, the official example runner is currently the reliable engine execution path, while `trtllm-bench` remains the reliable path for latency/throughput metrics.

## Quality Result

Quality checks ran against 25 deterministic prompts.

| Variant | Passed | Total | Pass rate |
| --- | ---: | ---: | ---: |
| HF baseline | 1 | 25 | 0.040 |
| TensorRT-LLM base | 1 | 25 | 0.040 |

This is not a TensorRT-specific quality regression. Both HF and TensorRT failed the same simple expected-substring checks because Qwen3 emitted `<think>` reasoning text and many deterministic prompts used very small `max_new_tokens`, so the answer was truncated before the expected substring.

Next quality fix:

- Add Qwen3 thinking-mode control in prompt formatting, preferably `enable_thinking=False` when supported by the tokenizer chat template.
- Alternatively increase deterministic prompt `max_new_tokens`, but disabling thinking is the cleaner benchmark control because the expected outputs are intentionally short.
- Rerun HF baseline and TensorRT prompt-set quality after that change.

## Parser/Script Changes Made After The Run

- `scripts/36_run_trtllm_prompt_set.py`
  - writes raw prompt outputs incrementally
  - supports `--resume`
  - supports `--examples-timeout-seconds`
  - supports `--thinking-mode`
- `scripts/aws/run_ec2_prompt_set.sh`
  - defaults to the confirmed `examples` backend
  - uses `--resume`
  - passes the example subprocess timeout
  - passes `LLM_THINKING_MODE`
- `scripts/90_run_all.sh`
  - uses the same prompt backend, resume, and timeout controls
  - passes `LLM_THINKING_MODE` to HF baseline and TensorRT prompt-set generation
- `src/hf_benchmark.py`
  - disables Qwen3 thinking mode by default in `thinking-mode=auto` by passing `enable_thinking=False` to the tokenizer chat template when supported
  - keeps the setting configurable with `--thinking-mode auto|on|off`
- `src/trtllm_result_parser.py` and `scripts/50_parse_results.py`
  - mark `examples` prompt-set rows as quality-output-only
  - do not claim HF-vs-TensorRT speedup for prompt-set rows that include per-prompt example-process startup
  - include a quality regression table in the generated final report

## Current Conclusion

The first GPU benchmark already provides valid TensorRT-LLM synthetic latency and throughput measurements. This prompt-set session adds TensorRT output capture for the same 50 prompts as HF, but direct speedup is still not claimed because the prompt-set TensorRT path used per-prompt example-process startup.

The next highest-value step is a short GPU rerun of HF baseline and TensorRT prompt-set quality with `LLM_THINKING_MODE=auto` or `off`, then regenerate quality/report files.
